- Cập nhật `Controllers/AccountController.cs` — thêm 2 actions:
  + `ChangePassword` GET `[Authorize]`
  + `ChangePassword` POST `[Authorize]` + `[ValidateAntiForgeryToken]`

### Logic POST (tóm tắt)

1. Validate `ModelState`
2. Lấy user hiện tại (`GetUserAsync`); null → `Challenge()`
3. Nếu `NewPassword == CurrentPassword` → lỗi “Mật khẩu mới không được trùng mật khẩu hiện tại”
4. `UserManager.ChangePasswordAsync(user, current, new)`
5. Fail `PasswordMismatch` → lỗi field current: “Mật khẩu hiện tại không chính xác”
6. Success → `RefreshSignInAsync` + `TempData` success + redirect PRG về GET

## Rủi ro phát hiện

- Navbar / Profile **chưa có link** sang `/Account/ChangePassword` — user phải gõ URL hoặc dùng link “Quay lại hồ sơ” (chỉ một chiều). Nên bổ sung link nhỏ ở Profile/navbar (ngoài scope T8).
- `RefreshSignInAsync` chỉ refresh **session hiện tại**. Session thiết bị khác có thể còn cookie đến khi Identity re-validate security stamp (interval mặc định) — chấp nhận được; nếu cần invalidate ngay mọi thiết bị thì gọi thêm `UpdateSecurityStampAsync` (hành vi mạnh hơn, giống Reset Password).
- Không sign-out sau đổi mật khẩu (khác Forgot/Reset) — đúng FRS UC-M1-08: chỉ thông báo thành công, user vẫn đang login.

## Hiểu được gì

- Đổi mật khẩu **đã login** dùng `ChangePasswordAsync` (cần mật khẩu cũ), **không** dùng `GeneratePasswordResetTokenAsync` như flow quên mật khẩu.
- Rule “mật khẩu mới ≠ mật khẩu cũ” nên check phía app (so sánh string sau khi form valid) vì Identity không luôn trả message rõ cho case này.
- Map `error.Code == "PasswordMismatch"` sang message tiếng Việt trên field `CurrentPassword` UX tốt hơn summary chung.
- PRG + `TempData` tránh resubmit form khi F5 sau khi đổi thành công.
